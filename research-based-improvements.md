# Wordy — План улучшений на основе исследования SLA

**Источник**: Анализ 800+ научных работ по Second Language Acquisition, данные Duolingo, EF EPI, FSI.

**Цель**: Превратить Wordy из приложения для запоминания слов в комплексный инструмент изучения английского, покрывающий все 4 направления модели Nation.

---

## Текущий баланс по модели Nation (4 направления)

| Направление (цель: 25%) | Сейчас | После плана |
|---|---|---|
| **Ввод (смысл)** — чтение, аудирование | ~5% | ~20% |
| **Выход (смысл)** — говорение, письмо | 0% | ~15% |
| **Языко-ориентированное** — грамматика, лексика | ~60% | ~40% |
| **Беглость** — лёгкие задания на скорость | ~35% | ~25% |

## Фаза 1: Контекст и глубокая обработка (2-3 недели)

**Научное обоснование**: Craik & Lockhart (1972) — семантическая обработка формирует устойчивый след в памяти. Nation (2001) — для запоминания нужно 8-12 осмысленных встреч со словом. Изолированные пары "слово-перевод" = поверхностная обработка.

### 1.1. Примеры предложений после ответа

**Что**: После правильного/неправильного ответа показывать 1-2 примера использования слова в предложении.

**Почему**: Каждый показ примера = +1 осмысленная встреча со словом. Переводит обработку с поверхностного уровня (структурного) на семантический (глубокий).

**Реализация**:

Сервер — расширить ответ `POST /api/quiz/answer-infinite`:
```
// В ответе на правильный/неправильный ответ добавить:
{
  ...existingFields,
  examples: [
    { en: "I need to make a decision", ru: "Мне нужно принять решение" },
    { en: "It was a difficult decision", ru: "Это было трудное решение" }
  ]
}
```

Данные: поле `examples` уже есть в `wordMeanings` (JSONB). Нужно:
1. В `quiz-service.ts` → `answerInfinite()`: добавить запрос examples для meaningId
2. Если examples пусто — не показывать блок
3. Максимум 2 примера (случайных из доступных)

Клиент — расширить feedback-экран:
1. В компоненте, который показывает результат ответа, добавить блок `<ExampleSentences />`
2. Дизайн: под feedback (correct/incorrect), мягкий фон, en жирным, ru — серым
3. Показывать 1.5-2 секунды перед следующим вопросом (или по тапу)

```
┌────────────────────────────────┐
│  ✓ Правильно!        +100 XP  │
│                                │
│  decision — решение            │
│                                │
│  ┌──────────────────────────┐  │
│  │ I need to make a decision│  │
│  │ Мне нужно принять решение│  │
│  └──────────────────────────┘  │
│  ┌──────────────────────────┐  │
│  │ It was a difficult one   │  │
│  │ Это было трудным         │  │
│  └──────────────────────────┘  │
└────────────────────────────────┘
```

**Файлы**:
- `server/src/services/quiz-service.ts` — добавить examples в ответ
- `client/src/components/game/example-sentences.tsx` — новый компонент
- Родительский компонент квиза — встроить ExampleSentences в feedback

**Оценка**: 1-2 дня

### 1.2. Cloze-тест (заполни пропуск в предложении)

**Что**: Новый тип вопроса — предложение с пропущенным словом, 4 варианта ответа.

**Почему**: Cloze test требует понимания контекста, а не просто распознавания пары слов. Это семантическая обработка (Craik & Lockhart). Задействует comprehension + выбор, активируя более глубокие когнитивные процессы.

**Пример**:
```
┌────────────────────────────────┐
│                                │
│  I need to _____ a decision    │
│                                │
│  ┌──────────┐  ┌──────────┐   │
│  │   make   │  │   do     │   │
│  └──────────┘  └──────────┘   │
│  ┌──────────┐  ┌──────────┐   │
│  │   take   │  │   get    │   │
│  └──────────┘  └──────────┘   │
│                                │
│  «принять решение»             │
└────────────────────────────────┘
```

**Реализация**:

Сервер — новый генератор `server/src/services/game/generators/cloze.ts`:
```ts
type ClozeQuestion = BaseQuestion & {
  type: 'cloze';
  sentence: string;        // "I need to _____ a decision"
  sentenceRu: string;      // "Мне нужно принять решение"
  blankPosition: number;   // индекс пропущенного слова
  options: string[];       // ["make", "do", "take", "get"]
  correctAnswer: string;   // "make"
};
```

Алгоритм генерации:
1. Получить meaning из SRS-пула (как обычно)
2. Взять пример предложения из `examples` поля meaning
3. Если примеров нет — fallback на другой генератор (return null)
4. Заменить целевое слово на `_____`
5. Сгенерировать 3 дистрактора:
   - Из той же части речи + похожей частотности
   - Приоритет: слова из той же коллекции пользователя (уже знакомые, но другие)
6. Проверка качества: дистракторы не должны подходить в контекст (базовая проверка: не синонимы целевого слова)

Фильтр применимости:
- Meaning должен иметь хотя бы 1 пример в `examples`
- Пример должен содержать целевое слово (или его формы)
- Предложение 4-12 слов (не слишком короткое, не слишком длинное)

Клиент — новый компонент `client/src/components/game/question-types/cloze.tsx`:
- Предложение сверху, пропуск выделен подчёркиванием
- Русский перевод предложения — мелким шрифтом под предложением
- Сетка 2×2 вариантов (переиспользовать стиль multiple-choice)
- Тот же feedback flow

Ротация:
- Добавить `'cloze'` в `GeneratorType` union
- В `pickGenerator()` — weight 1.0 (наравне с остальными)
- Условие применимости: meaning имеет examples (проверяется при выборе генератора)

**Файлы**:
- `server/src/services/game/types.ts` — добавить ClozeQuestion, 'cloze' в GeneratorType
- `server/src/services/game/generators/cloze.ts` — новый генератор
- `server/src/services/game/generators/index.ts` — реэкспорт
- `server/src/services/quiz-service.ts` — интеграция в ротацию
- `server/src/routes/quiz.ts` — добавить 'cloze' в VALID_GENERATORS
- `client/src/types/game.ts` — ClozeQuestion тип
- `client/src/components/game/question-types/cloze.tsx` — UI компонент
- Родительский компонент квиза — добавить case 'cloze'

**Оценка**: 3-4 дня

### 1.3. Контекстная подсказка при ошибке

**Что**: При неправильном ответе показывать не только правильный перевод, но и пример предложения с этим словом.

**Почему**: Момент ошибки — окно повышенного внимания (surprise effect). Показ контекста в этот момент значительно улучшает запоминание (Kornell et al., 2009).

**Реализация**: Уже покрывается пунктом 1.1 (примеры после ответа). Просто убедиться, что при ошибке примеры показываются **обязательно** (а при правильном — опционально, с возможностью скрыть).

**Оценка**: 0 дней (входит в 1.1)

## Фаза 2: Аудирование (1-2 недели)

**Научное обоснование**: Ellis (2008) — "автоматизация одного навыка (аудирования) напрямую не помогает автоматизации другого (говорения)". Аудирование — отдельный процедурный навык. Nation — 25% времени на "ввод, ориентированный на смысл". У русскоговорящих говорение = самый слабый навык (EF EPI: 462 vs 536 за чтение), но аудирование — фундамент для говорения.

### 2.1. Аудио-вопрос (слушай → выбери перевод)

**Что**: Новый тип вопроса — проигрывается произношение слова (TTS), нужно выбрать правильный перевод. Текст слова НЕ показывается.

**Почему**: Тренирует распознавание на слух — отдельный навык от чтения. Заставляет формировать звуковой образ слова, а не только визуальный.

**Пример**:
```
┌────────────────────────────────┐
│                                │
│         🔊 [Play]              │
│     (проигрывается "decision") │
│                                │
│  Что вы услышали?              │
│                                │
│  ┌──────────┐  ┌──────────┐   │
│  │ решение  │  │ деление  │   │
│  └──────────┘  └──────────┘   │
│  ┌──────────┐  ┌──────────┐   │
│  │ движение │  │ видение  │   │
│  └──────────┘  └──────────┘   │
│                                │
└────────────────────────────────┘
```

**Реализация**:

Сервер — новый генератор `server/src/services/game/generators/listening.ts`:
```ts
type ListeningQuestion = BaseQuestion & {
  type: 'listening';
  audioWord: string;           // слово для TTS (на клиенте)
  transcription: string;       // показать после ответа
  options: string[];           // варианты переводов (русские)
  correctAnswer: string;       // правильный перевод
};
```

Алгоритм:
1. Meaning из SRS-пула
2. Направление всегда en→ru (слушаем английское, выбираем русский перевод)
3. Дистракторы: переводы других слов (та же логика, что в multiple-choice en→ru)
4. Отправляем `audioWord` — клиент сам проигрывает через TTS API

TTS на клиенте:
- Уже используется `speechSynthesis` API (существующая кнопка произношения)
- Для listening-вопроса: автоматическое воспроизведение при появлении вопроса
- Кнопка "Повторить" для повторного прослушивания (максимум 3 раза за вопрос)

Клиент — `client/src/components/game/question-types/listening.tsx`:
- Большая кнопка-динамик в центре (auto-play при загрузке)
- Кнопка повторного воспроизведения (счётчик оставшихся прослушиваний)
- Сетка 2×2 русских переводов
- После ответа — показать текст слова + транскрипцию

Ротация:
- `'listening'` в GeneratorType
- Применим всегда (у всех слов есть текст для TTS)
- Частота: ~15% вопросов (каждый 6-7й)

**Файлы**:
- `server/src/services/game/types.ts` — ListeningQuestion
- `server/src/services/game/generators/listening.ts` — генератор
- `server/src/services/quiz-service.ts` — ротация
- `client/src/types/game.ts` — тип
- `client/src/components/game/question-types/listening.tsx` — UI
- Родительский компонент квиза — case 'listening'

**Оценка**: 2-3 дня

### 2.2. Аудио-диктант (слушай → напиши)

**Что**: Проигрывается слово, нужно его набрать на клавиатуре. Комбинация аудирования + spelling.

**Почему**: Активирует сразу два навыка — распознавание на слух + продуцирование (письменное). Это recall, не recognition — значительно более эффективно для запоминания (Karpicke & Roediger, 2008: +100% удержание).

**Пример**:
```
┌────────────────────────────────┐
│                                │
│         🔊 [Play]              │
│     (проигрывается "decision") │
│                                │
│  Напишите услышанное слово:    │
│                                │
│  ┌──────────────────────────┐  │
│  │ decisio█                 │  │
│  └──────────────────────────┘  │
│                                │
│       [Проверить]              │
│                                │
│  «решение»  — подсказка       │
└────────────────────────────────┘
```

**Реализация**:

Это реализация `text-input` типа вопроса, который уже определён в типах, но не реализован.

Сервер — новый генератор `server/src/services/game/generators/dictation.ts`:
```ts
type DictationQuestion = BaseQuestion & {
  type: 'dictation';
  audioWord: string;           // слово для TTS
  hint: string;                // перевод (подсказка)
  correctAnswer: string;       // правильное написание
  acceptableAnswers: string[]; // допустимые варианты (с/без дефиса и т.д.)
};
```

Алгоритм:
1. Meaning из SRS-пула
2. Фильтр: слово >= 4 букв (как spelling)
3. `acceptableAnswers`: основная форма + лемма (если отличается)
4. Нечёткое сравнение: Levenshtein distance <= 1 для слов > 6 букв (прощаем 1 опечатку)

Клиент — `client/src/components/game/question-types/dictation.tsx`:
- Кнопка воспроизведения (auto-play)
- Текстовое поле ввода
- Подсказка (русский перевод) — показана сразу, мелким шрифтом
- Кнопка "Проверить" или Enter
- Feedback: если правильно — зелёный, если почти (Levenshtein 1) — жёлтый с показом правильного, если неправильно — красный

**Файлы**:
- `server/src/services/game/types.ts` — DictationQuestion
- `server/src/services/game/generators/dictation.ts` — генератор
- `client/src/types/game.ts` — тип
- `client/src/components/game/question-types/dictation.tsx` — UI
- `client/src/lib/levenshtein.ts` — функция расстояния Левенштейна (простая, ~15 строк)

**Оценка**: 3-4 дня

### 2.3. Минимальные пары (фонетика для русскоговорящих)

**Что**: "Что вы слышали: ship или sheep?" — тренировка различения звуков, проблемных для русскоговорящих.

**Почему**: Исследование выделяет конкретные фонетические барьеры: /ɪ/ vs /iː/ (ship/sheep), /θ/ vs /s/ (think/sink), /w/ vs /v/ (wine/vine). Целенаправленная работа над ними даёт быстрый эффект.

**Решение**: Отложить до фазы 4+. Требует отдельного набора данных (минимальные пары), высокого качества TTS и тонкой настройки. Пока покрывается listening-вопросами (2.1) на общем уровне.

## Фаза 3: Свободное воспроизведение / Recall (1-2 недели)

**Научное обоснование**: Karpicke & Roediger (2008) — повторное тестирование через recall улучшает удержание на 100%+ по сравнению с повторным изучением. Ключевое различие recognition vs recall: выбрать из 4 вариантов ≠ вспомнить самому.

### 3.1. Free Recall (напиши перевод без вариантов)

**Что**: Показывается английское слово, нужно самому написать русский перевод (без вариантов). Или наоборот: русское → напиши английское.

**Почему**: Это настоящий recall — самый эффективный способ закрепления по данным исследований. Все текущие типы вопросов в Wordy — recognition (выбор из готовых вариантов).

**Пример**:
```
┌────────────────────────────────┐
│                                │
│         decision               │
│       /dɪˈsɪʒ.ən/      🔊    │
│                                │
│  Введите перевод:              │
│                                │
│  ┌──────────────────────────┐  │
│  │ решение█                 │  │
│  └──────────────────────────┘  │
│                                │
│       [Проверить]              │
└────────────────────────────────┘
```

**Реализация**:

Сервер — новый генератор `server/src/services/game/generators/free-recall.ts`:
```ts
type FreeRecallQuestion = BaseQuestion & {
  type: 'free-recall';
  direction: 'en-ru' | 'ru-en';
  prompt: string;              // слово/перевод для показа
  transcription?: string;      // только для en→ru
  acceptableAnswers: string[]; // все допустимые ответы
};
```

Алгоритм:
1. Meaning из SRS-пула
2. Направление: 50/50 en→ru / ru→en
3. `acceptableAnswers` для en→ru:
   - Основной перевод meaning
   - Синонимы из поля `synonyms`
   - Переводы других meanings с тем же словом (другие значения)
4. `acceptableAnswers` для ru→en:
   - Текст слова
   - Лемма (если отличается)
5. Нечёткое сравнение:
   - Точное совпадение (case-insensitive, trim)
   - Допуск буквы "ё" ↔ "е" для русских ответов
   - Levenshtein <= 1 для слов > 5 букв

Условие применимости:
- **Только для слов с SRS stage >= 2** (уже частично выученных)
- На stage 0-1 слово ещё новое — recall слишком сложен, демотивирует
- Это ключевое решение: recall подключается как "усиление" уже начатого обучения

Клиент — `client/src/components/game/question-types/free-recall.tsx`:
- Слово + транскрипция + кнопка произношения (для en→ru)
- Или русский перевод (для ru→en)
- Текстовое поле с автофокусом
- Кнопка "Проверить" (+ Enter)
- Feedback: точное совпадение → зелёный, почти → жёлтый ("Почти! Правильно: ..."), неверно → красный + правильный ответ

Ротация:
- `'free-recall'` в GeneratorType
- Условие: srsStage >= 2
- Частота: ~10-15% вопросов (для подходящих слов)

**Файлы**:
- `server/src/services/game/types.ts` — FreeRecallQuestion
- `server/src/services/game/generators/free-recall.ts` — генератор
- `client/src/types/game.ts` — тип
- `client/src/components/game/question-types/free-recall.tsx` — UI
- `client/src/lib/answer-check.ts` — утилита нечёткого сравнения (Levenshtein + ё/е + trim)

**Оценка**: 3-4 дня

## Фаза 4: Грамматика для русскоговорящих (2-3 недели)

**Научное обоснование**: Артикли — проблема №1 для русскоговорящих (в русском их нет). Present Perfect не имеет прямого эквивалента. Ложные друзья создают иллюзию понимания. Грамматико-переводной метод (80% теории) уничтожает коммуникативную компетенцию — нужна **практика через квизы**, не справочник.

### 4.1. Коллекция "Ложные друзья переводчика"

**Что**: Системная коллекция из 100+ пар слов-ловушек для русскоговорящих (magazine ≠ магазин, accurate ≠ аккуратный).

**Почему**: Словарь falsefriends.ru содержит 1000+ пар. Ложные друзья — одна из самых коварных ловушек для русскоговорящих, потому что общее латинско-греческое наследие создаёт иллюзию понимания.

**Реализация**:

1. Создать seed-данные: `server/src/data/false-friends.ts`
   - 100 наиболее опасных пар
   - Структура: `{ word: string, wrongRu: string, correctRu: string, example: string }`
   - Примеры: magazine (периодика, не магазин), accurate (точный, не аккуратный), actual (реальный, не актуальный)

2. Системная коллекция в seed:
   - Категория: "Ловушки" (или новая категория "Для русскоговорящих")
   - CEFR: A2-B1 (большинство ложных друзей актуальны на этом уровне)
   - Слова добавить из существующей БД (если есть) или как новые

3. Специальный тип вопроса (или адаптация multiple-choice):
   - Показ: "magazine" → варианты: "журнал" / "магазин" / "склад" / "витрина"
   - Ложный друг ("магазин") всегда среди вариантов как ловушка
   - После ответа: пояснение "Magazine ≠ магазин! Magazine = журнал, периодика"

**Файлы**:
- `server/src/data/false-friends.ts` — данные
- Seed-скрипт или startup-migration для создания коллекции
- Возможно: специальная пометка в `wordMeanings` для показа warning ("ложный друг")

**Оценка**: 3-4 дня

### 4.2. Грамматика: квиз по артиклям (a / the / —)

**Что**: Предложение с пропуском — выбери правильный артикль (a, an, the, или без артикля).

**Почему**: Артикли — проблема №1 для русскоговорящих. В русском языке артиклей нет вообще. Ошибки сохраняются даже на C1. Квиз-формат (не теория) — единственный способ выработать интуицию через практику.

**Пример**:
```
┌────────────────────────────────┐
│                                │
│   I saw _____ dog in _____ park│
│                                │
│  ┌────┐ ┌────┐ ┌────┐ ┌────┐  │
│  │ a  │ │ an │ │ the│ │ — │   │
│  └────┘ └────┘ └────┘ └────┘  │
│                                │
│  Пропуск 1 из 2               │
└────────────────────────────────┘
```

**Реализация**:

Данные — `server/src/data/article-exercises.ts`:
```ts
type ArticleExercise = {
  id: number;
  sentence: string;          // "I saw ___ dog in ___ park"
  blanks: ArticleBlank[];    // позиции пропусков
  difficulty: 1 | 2 | 3;    // 1=базовые правила, 2=исключения, 3=нюансы
  rule: string;              // "Первое упоминание = a/an, повторное = the"
  ruleCategory: string;      // "first_mention", "unique", "zero_article"
};

type ArticleBlank = {
  position: number;
  correctAnswer: 'a' | 'an' | 'the' | 'zero';  // zero = без артикля
  explanation: string;       // "Первое упоминание собаки → a"
};
```

Начальный набор: ~150 предложений, покрывающих основные правила:
- Первое/повторное упоминание (a → the)
- Уникальные объекты (the sun, the moon)
- Нулевой артикль (неисчисляемые, абстрактные)
- a vs an (гласный звук, не буква: a university, an hour)
- Устойчивые выражения (go to school, at home)

API: интеграция в grammar-модуль:
- `POST /api/grammar/articles/next` — следующее упражнение
- `POST /api/grammar/articles/answer` — проверка
- Прогрессия: от difficulty 1 к 3 (SRS-подобная, но по правилам, а не по словам)

Клиент — `client/src/components/grammar/article-quiz.tsx`:
- Предложение с подсвеченным пропуском
- 4 кнопки: a / an / the / — (без артикля)
- Если несколько пропусков — последовательно (1 из N)
- После ответа: пояснение правила (зелёный/красный фон)
- Новый таб на странице /grammar: "Артикли"

**Файлы**:
- `server/src/data/article-exercises.ts` — 150+ предложений
- `server/src/services/grammar/article-service.ts` — логика квиза
- `server/src/routes/grammar.ts` — новые endpoints
- `client/src/components/grammar/article-quiz.tsx` — UI
- `client/src/stores/grammar-store.ts` — расширить стор

**Оценка**: 5-7 дней

### 4.3. Грамматика: квиз по временам в контексте

**Что**: Расширение существующего модуля времён — от справочника к практике. Предложение + контекст → выбери правильное время.

**Почему**: Справочник времён уже есть (tense grid). Но знание правил (декларативное) ≠ умение применять (процедурное). Модель ACT Андерсона: переход от "знаю что" к "знаю как" происходит только через практику.

**Пример**:
```
┌────────────────────────────────┐
│                                │
│  She _____ a book when I       │
│  called her yesterday.         │
│                                │
│  ┌────────────────────────┐    │
│  │ reads                  │    │
│  ├────────────────────────┤    │
│  │ was reading        ✓   │    │
│  ├────────────────────────┤    │
│  │ has read               │    │
│  ├────────────────────────┤    │
│  │ had been reading       │    │
│  └────────────────────────┘    │
│                                │
│  Подсказка: действие в процессе│
│  в прошлом, прерванное другим  │
└────────────────────────────────┘
```

**Реализация**:

Данные — `server/src/data/tense-exercises.ts`:
```ts
type TenseExercise = {
  id: number;
  sentence: string;              // "She _____ a book when I called"
  sentenceRu: string;            // "Она читала книгу, когда я позвонил"
  options: TenseOption[];        // варианты с указанием времени
  correctTense: string;          // "past_continuous"
  signalWords: string[];         // ["when", "yesterday"]
  explanation: string;           // "Действие в процессе, прерванное другим"
  difficulty: 1 | 2 | 3;
};

type TenseOption = {
  text: string;                  // "was reading"
  tenseKey: string;              // "past_continuous"
};
```

Начальный набор: ~100 предложений, по ~8 на каждое из 12 времён + смешанные.

**Файлы**:
- `server/src/data/tense-exercises.ts` — данные
- `server/src/services/grammar/tense-quiz-service.ts` — логика
- `server/src/routes/grammar.ts` — endpoints
- `client/src/components/grammar/tense-quiz.tsx` — UI
- Страница /grammar — новый таб "Квиз времён"

**Оценка**: 4-5 дней

## Фаза 5: Коллокации и уровневая адаптация (2-3 недели)

**Научное обоснование**: Thornbury (2002) — "способность использовать широкий диапазон лексических чанков — вот что отличает продвинутого ученика от промежуточного". Richards (2008) — промежуточное плато — прогресс становится невидимым.

### 5.1. Коллокации (устойчивые сочетания)

**Что**: Новый тип контента — не отдельные слова, а устойчивые сочетания: make a decision (не do a decision), heavy rain (не strong rain), strong coffee (не powerful coffee).

**Почему**: Это главный барьер на уровне B1+ → B2. Ученик знает слова "make" и "decision" по отдельности, но не знает, что они идут вместе. Именно это отличает "intermediate" от "advanced".

**Реализация**:

Данные — новая сущность `collocations`:
```ts
// server/src/data/collocations.ts
type Collocation = {
  id: number;
  phrase: string;           // "make a decision"
  translationRu: string;   // "принять решение"
  baseWord: string;         // "decision" (ключевое слово)
  collocate: string;        // "make" (сочетающееся слово)
  wrongCollocates: string[]; // ["do", "take", "have"] — типичные ошибки
  cefrLevel: string;        // "B1"
  category: string;         // "verb+noun", "adj+noun", "adv+adj"
  example: string;          // "She made a difficult decision"
};
```

Типы вопросов (встраиваются в существующие генераторы):
1. **Выбери правильное сочетание**: "_____ a decision" → make / do / take / have
2. **Найди ошибку**: "She did a mistake" — что неправильно? → "did" (правильно: "made")

Коллекции: системные коллекции "Коллокации B1", "Коллокации B2".

Начальный набор: 200 самых частых коллокаций, сгруппированных по шаблонам:
- Verb + Noun: make/do/take/have + существительное (100 шт.)
- Adj + Noun: heavy rain, strong coffee (50 шт.)
- Adv + Adj: deeply concerned, highly unlikely (50 шт.)

**Файлы**:
- `server/src/db/schema.ts` — таблица collocations (или TS-константы на MVP)
- `server/src/data/collocations.ts` — данные
- `server/src/services/game/generators/collocation.ts` — генератор вопросов
- `client/src/components/game/question-types/collocation.tsx` — UI
- Seed для системных коллекций

**Оценка**: 5-7 дней

### 5.2. Визуализация прогресса по CEFR

**Что**: На странице профиля показать прогресс-бары по CEFR-уровням: "A1: 87% слов выучено, A2: 34%, B1: 12%".

**Почему**: На уровне B1-B2 прогресс становится невидимым (Richards, 2008). Визуализация по CEFR делает прогресс видимым и создаёт чёткую цель ("дойти до B1").

**Реализация**:

Сервер — новый endpoint:
```
GET /api/users/me/cefr-progress
→ {
    levels: [
      { level: "A1", totalWords: 465, learnedWords: 405, percent: 87 },
      { level: "A2", totalWords: 820, learnedWords: 279, percent: 34 },
      { level: "B1", totalWords: 1200, learnedWords: 144, percent: 12 },
      ...
    ]
  }
```

Логика:
- `totalWords`: COUNT слов в системной коллекции этого уровня
- `learnedWords`: COUNT слов где userWordProgress.srsStage >= 3 (mastered)
- Группировка по CEFR-уровню коллекции

Клиент — компонент `CefrProgress`:
- Вертикальный список уровней A1-C1
- Каждый: название + прогресс-бар + "405/465"
- Текущий активный уровень подсвечен
- Размещение: страница профиля или отдельная вкладка

**Файлы**:
- `server/src/routes/users.ts` — endpoint
- `server/src/services/user-service.ts` — логика подсчёта
- `client/src/components/profile/cefr-progress.tsx` — UI

**Оценка**: 2-3 дня

## Фаза 6: Элементы говорения (2-3 недели)

**Научное обоснование**: Swain (1985) — Output Hypothesis: понятный ввод необходим, но недостаточен. Говорение выполняет 4 функции: замечание пробелов, проверка гипотез, метаязыковая рефлексия, развитие автоматизма. Систематический обзор 2025 года подтверждает шэдоуинг для улучшения произношения и беглости.

### 6.1. Повтори вслух (Shadowing Light)

**Что**: После правильного ответа — опциональное предложение произнести слово вслух. Без оценки, без микрофона. Просто промпт "Повтори: decision /dɪˈsɪʒ.ən/" с кнопкой воспроизведения образца.

**Почему**: Минимальная форма output-практики. Не требует распознавания речи. Снижает тревожность (нет оценки). Шэдоуинг — подтверждённая техника даже в простейшей форме.

**Реализация**:

Клиент — расширение feedback-экрана:
1. После correct feedback + примеров (1.1) — опциональный блок:
   ```
   ┌──────────────────────────────┐
   │  🔊 Повтори: decision       │
   │     /dɪˈsɪʒ.ən/             │
   │                              │
   │  [Послушать]  [Пропустить]   │
   └──────────────────────────────┘
   ```
2. Кнопка "Послушать" — TTS воспроизведение
3. Не блокирует переход к следующему вопросу (можно пропустить)
4. Настройка в settings: вкл/выкл "Повтори вслух" (по умолчанию: выкл)

Серверная часть: не требуется (всё на клиенте).

**Файлы**:
- `client/src/components/game/shadowing-prompt.tsx` — компонент
- Родительский компонент feedback — встроить
- `client/src/stores/user-store.ts` — настройка вкл/выкл

**Оценка**: 1-2 дня

### 6.2. Произношение с распознаванием (v2)

**Что**: Используя Web Speech API (`SpeechRecognition`), оценить произношение пользователя. Показать, насколько близко к образцу.

**Почему**: Обратная связь критически важна для коррекции произношения. Без неё фоссилизируются фонетические ошибки (исследование: ошибки становятся "практически неискоренимыми").

**Ограничения**:
- Web Speech API доступен не во всех WebView (проверить в Telegram)
- Качество распознавания сильно зависит от устройства и окружения
- Нужен fallback (6.1) если API недоступен

**Реализация**: Отложить до подтверждения работоспособности Web Speech API в Telegram WebView. Сначала выпустить 6.1 (без микрофона), потом A/B тест с распознаванием.

**Оценка**: 3-5 дней (если API работает), ∞ (если нет)

## Фаза 7: Мотивация и борьба с оттоком (1-2 недели)

**Научное обоснование**: 42% бросивших назвали потерю мотивации главной причиной (TheKnowledgeAcademy). Dörnyei — мотивация динамична, её нужно поддерживать. Alamer et al. (2025) — рост навыков снижает тревожность (позитивная обратная связь).

### 7.1. Мотивационные milestones и сообщения

**Что**: Milestone-уведомления при достижении значимых точек + мотивационные сообщения при длинных стриках.

**Примеры milestones**:
- "Ты знаешь 100 слов! Этого достаточно для базового общения"
- "300 слов — ты понимаешь ~65% повседневной речи"
- "500 слов — уровень A1 почти пройден!"
- "1000 слов — ты в числе 20% лучших изучающих!"
- "A1 пройден! Переход к A2"

**Примеры streak-сообщений**:
- 7 дней: "Неделя без перерыва! Привычка формируется"
- 14 дней: "Две недели! Исследования показывают: 21 день = привычка"
- 30 дней: "Месяц! Ты в топ-5% по стабильности обучения"
- 100 дней: "100 дней! Это уровень профессионала"

**Реализация**:

Данные — `server/src/config/milestones-config.ts`:
```ts
type Milestone = {
  id: string;
  type: 'words_learned' | 'streak_days' | 'cefr_level' | 'total_answers';
  threshold: number;
  title: string;
  description: string;
  reward?: { gems: number };
};
```

Логика:
- Проверка milestones при каждом ответе (в progression-service)
- Если milestone достигнут и ещё не показан → вернуть в ответе
- Хранение показанных milestones: новое поле `shownMilestones` в users (JSONB array)

Клиент:
- Modal/toast при получении milestone в ответе API
- Анимация (Lottie или простой CSS)
- Кнопка "Круто!" для закрытия

**Файлы**:
- `server/src/config/milestones-config.ts` — конфигурация
- `server/src/services/progression-service.ts` — проверка milestones
- `server/src/db/schema.ts` — поле shownMilestones (или отдельная таблица)
- `client/src/components/milestone-modal.tsx` — UI

**Оценка**: 3-4 дня

### 7.2. Адаптивная сложность

**Что**: Если пользователь ошибается > 50% за последние 10 вопросов — временно снизить сложность (только знакомые слова, только multiple-choice). Если < 20% ошибок — повысить (cloze, free-recall, новые слова).

**Почему**: Крашен (1982) — аффективный фильтр: "низкая самооценка и тревожность создают ментальный блок". Серия ошибок → фрустрация → бросание. Адаптивная сложность поддерживает зону "i+1" (чуть сложнее текущего уровня).

**Реализация**:

Трекинг: последние N ответов (уже есть в session data).
Логика в quiz-service:
- `errorRate > 0.5` за последние 10 → "easy mode": только review (знакомые слова), только multiple-choice
- `errorRate < 0.2` за последние 10 → "challenge mode": больше cloze, free-recall, новых слов
- Нормальный режим: текущая логика

Без UI-индикации (пользователь не должен знать о переключении — это должно ощущаться естественно).

**Файлы**:
- `server/src/services/quiz-service.ts` — логика адаптации
- Трекинг: уже есть данные об ответах в сессии

**Оценка**: 2-3 дня

## Фаза 8: Экстенсивное чтение (3-4 недели, v2)

**Научное обоснование**: Nation назвал включение экстенсивного чтения "самым эффективным изменением, которое преподаватель может внести в курс". Мета-анализ 2025 года подтвердил эффекты во всех языковых доменах.

### 8.1. Мини-тексты с comprehension questions

**Что**: Короткие адаптированные тексты (3-7 предложений) на уровне пользователя. После текста — 2-3 вопроса на понимание. Знакомые слова подсвечены.

**Почему**: Comprehensible input (i+1) — фундаментальная гипотеза Крашена. Экстенсивное чтение работает, потому что создаёт множественные осмысленные встречи со словами в контексте.

**Концепция** (детальная реализация — отдельный документ):

Данные:
- Тексты генерируются с помощью LLM (GPT/Claude) и ревьюятся вручную
- Уровень текста привязан к CEFR: A1 тексты используют только A1 слова + 1-2 новых
- Тематические категории: повседневная жизнь, путешествия, работа, хобби
- ~50 текстов на уровень для запуска

Формат:
```
┌────────────────────────────────┐
│  📖 Уровень A2                │
│                                │
│  Yesterday I went to a new     │
│  café near my house. The       │
│  coffee was really good, but   │
│  the prices were quite high.   │
│  I decided to come back next   │
│  week with my friends.         │
│                                │
│  [decided] [prices] — подсвечены│
│  как изучаемые слова            │
│                                │
│  ─────────────────────────────  │
│  Вопрос 1 из 2:               │
│  Where did the person go?      │
│  ○ To a restaurant             │
│  ○ To a café              ✓    │
│  ○ To a shop                   │
│  ○ To a park                   │
└────────────────────────────────┘
```

**Оценка**: 3-4 недели (включая создание контента)

12. **Коллокации** (5.1) — прорыв плато B1→B2

### Спринт 6+ (3-4 недели): Экстенсивное чтение
14. **Мини-тексты** (8.1) — comprehensible input

## Что НЕ включено в план (и почему)

| Идея | Причина исключения |
|------|-------------------|
| AI-диалоги (ChatGPT-like) | Слишком амбициозно, требует LLM API, высокая стоимость |
| Минимальные пары (/θ/ vs /s/) | Требует высококачественного TTS, отдельного набора данных |
| Sentence Builder (конструктор фраз) | Уже спроектирован в grammar-module-design.md, реализовать после фазы 4 |
| Tense Transform (трансформация времён) | Аналогично — после grammar фазы |
| Произношение с микрофоном | Web Speech API может не работать в Telegram WebView |
| Полноценные тексты/книги | Нужна инфраструктура контента, начать с мини-текстов |
